# noon 批量运营与上架前检查设计

## 背景

当前 `noon-tools` 已经具备 1688 采集、本地商品仓库、AI 生成 noon 属性、noon 上传、批量更新表导出和本地 UI。截图里的功能覆盖多平台采集、翻译、合规、利润、批量运营、多店铺、仓位预约、选品、图搜和中文导航，范围过大，不适合作为一个实施计划。

第一期聚焦现有链路中最短、最可验证的闭环：在商品采集后、上传前、上传后提供批量运营和上架前检查能力。

## 目标

构建 `noon 批量运营与上架前检查`：让用户能在本地商品仓库中批量检查商品、计算利润、调整价格和库存、停售商品，并为多店铺上传状态打基础。

成功标准：

- 商品仓库页能筛选并多选商品或 SKU。
- 系统能对选中商品运行合规检查和利润计算。
- 阻塞问题会阻止上传和导出；警告问题允许继续但必须展示。
- 用户能批量改价、改库存、停售商品。
- 批量操作能更新本地 noon 属性或运营状态，并生成 noon 批量更新表。
- 上传状态按店铺隔离，支持后续多店铺运营。

## 非目标

第一期不实现以下能力：

- 拼多多、Amazon、Noon 商品采集。
- FBN 7x24 自动抢仓或自动预约。
- 图搜商品。
- 选品助手。
- 完整 noon 后台中文化。
- 直接自动删除 noon 后台商品。
- 云同步、团队权限、远程数据库。

## 现有约束

- 商品主数据继续使用 `products/<platform>/<repository>/<productId>/`。
- 当前平台继续以 `1688` 为主。
- `meta.json` 保存来源事实和本地素材事实。
- `noon-product-attributes.json` 保存 noon 目标商品数据。
- 上传和导出默认只处理通过本地技术校验的商品。
- `index.json` 是派生文件，不作为唯一事实来源。
- 不引入外部数据库；继续使用本地 JSON 和 xlsx 导出。

## 功能设计

### 1. 商品运营状态

每个商品或 SKU 需要有本地运营状态，用于区分可售、停售和待处理。

建议状态：

- `active`: 可继续运营。
- `inactive`: 本地标记停售，批量库存应为 0。
- `needs_review`: 需要人工确认，不应自动上传。

状态可以保存在 `noon-product-attributes.json` 的商品级或 variant 级字段中。第一期优先支持商品级状态；variant 级状态只在现有结构已经容易表达时实现。

### 2. 多店铺上传状态基础

上传状态必须按店铺隔离，避免一个商品在 A 店铺已上传却影响 B 店铺。

建议结构：

```json
{
  "stores": {
    "noon-uae-main": {
      "status": "uploaded",
      "uploadedAt": "2026-06-18T00:00:00.000Z"
    },
    "noon-sa-second": {
      "status": "not_uploaded"
    }
  }
}
```

第一期只需要支持：

- 配置多个店铺。
- 每个店铺有名称、noon Add Product URL、浏览器 profile 标识。
- 商品上传状态按店铺读取和写入。
- UI 可以按店铺过滤上传状态。

### 3. 利润计算

利润计算不应直接覆盖人工售价。系统先生成建议值，用户确认后才写入 noon 属性或导出表。

输入：

- 商品成本 CNY。
- 国内或跨境运费 CNY。
- 汇率。
- noon 平台费率。
- 目标利润率。
- 当前或目标售价 AED。

输出：

- 建议售价 AED。
- 预估利润 AED。
- 毛利率。
- 是否低于目标利润率。

利润结果写入商品的运营检查结果中，方便 UI 展示和后续导出。

### 4. 合规检测

合规检测作为上架前 gate。它不负责修复内容，只负责指出问题。

检查项：

- 标题存在且长度符合 noon 字段限制。
- 英文标题和阿拉伯语标题存在。
- 类目字段存在。
- 禁用词检查。
- 图片数量满足最低上传要求。
- SKU 存在且本地唯一。
- barcode 存在且本地唯一。
- 价格、库存、尺寸、重量存在。
- 引用的本地图片文件存在。

结果分两类：

- `blockingIssues`: 阻止上传和导出。
- `warnings`: 允许继续，但 UI 必须展示。

示例：

```json
{
  "blockingIssues": [
    {
      "code": "missing_barcode",
      "message": "variant 缺少 barcode"
    }
  ],
  "warnings": [
    {
      "code": "low_margin",
      "message": "毛利率低于目标利润率"
    }
  ]
}
```

### 5. 批量运营

批量运营以仓库页为入口，支持多选商品或 SKU 后执行。

第一期操作：

- 批量改价：写入确认后的价格，并生成 `Global Price Update`。
- 批量改库存：写入库存，并生成 `Stock Import`。
- 批量停售：将库存设为 0，并将本地运营状态设为 `inactive`。
- 批量改时效：第一期只保存本地运营字段，不写入 xlsx。

每次批量操作都需要在执行前展示：

- 影响商品数。
- 影响 SKU 数。
- 会写入哪些本地文件。
- 会生成哪些导出文件。
- 是否存在阻塞问题。

### 6. UI 设计

商品仓库页增加批量运营工具栏。

筛选维度：

- 仓库。
- 商品状态。
- 合规状态。
- 上传状态。
- 店铺。

操作入口：

- 运行检查。
- 计算利润。
- 批量改价。
- 批量改库存。
- 批量停售。
- 生成 noon 批量更新表。

操作结果：

- 展示成功数量、跳过数量、失败数量。
- 展示阻塞原因和警告。
- 提供 xlsx 下载链接。
- 将最近操作日志展示在页面中。

## 数据流

```text
商品仓库页
  -> GET /api/products
  -> 选择仓库/商品/SKU
  -> POST /api/operation-checks
  -> 读取 meta.json + noon-product-attributes.json
  -> 返回 blockingIssues / warnings / profit
  -> 用户确认批量操作
  -> POST /api/bulk-operations
  -> 更新 noon-product-attributes.json 或上传状态
  -> 生成 noon 批量更新表
  -> UI 展示结果和下载链接
```

## 错误处理

- 商品目录缺少 `meta.json` 时，该商品标记为阻塞，不中断整个批量任务。
- 商品目录缺少 `noon-product-attributes.json` 时，该商品标记为阻塞，不中断整个批量任务。
- SKU 或 barcode 冲突时，相关商品阻止上传和导出。
- xlsx 生成失败时，不回滚已经确认写入的本地 JSON，但必须在操作结果中报告失败。
- 用户取消或任务中断时，已完成的商品保留结果，未处理商品保持原状。

## 测试策略

单元测试：

- 利润计算。
- 合规检测。
- 批量操作输入校验。
- 多店铺上传状态读取和写入。
- 批量导出时跳过 blocking 商品。

集成测试：

- 使用 fixtures 构造多个商品、多个 SKU、多个店铺。
- 验证批量改价后导出的 `Global Price Update` 内容。
- 验证批量改库存和停售后导出的 `Stock Import` 内容。
- 验证有阻塞问题的商品不会进入上传或导出。

手动验证：

- 启动 `npm run ui`。
- 打开商品仓库页。
- 选择一个仓库运行检查。
- 对部分商品执行批量改价、改库存、停售。
- 下载并检查生成的 xlsx。

## 推荐实施顺序

1. 增加合规检查和利润计算的纯函数模块。
2. 增加批量运营数据模型和本地 JSON 写入逻辑。
3. 扩展 noon 批量更新导出，让它能消费检查结果和运营状态。
4. 增加多店铺上传状态基础。
5. 扩展 `scripts/server.js` API。
6. 扩展 `public/index.html` 的商品仓库页 UI。
7. 补齐单元测试和端到端手动验证文档。

## 第一期默认决策

- 批量改时效第一期只写入本地运营字段；不进入现有 noon xlsx 模板。
- 店铺 ID 由用户手动输入，系统只做安全路径字符校验和重复校验。
- 利润计算的平台费率按店铺配置；未选择店铺时使用全局默认费率。
