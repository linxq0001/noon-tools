# Noon 商品目录双同步方案设计

## 目标

在 `/noon-workbench` 中同时提供“API 快速同步”和“导出同步”两个明确入口。用户主动选择同步来源；两种来源最终写入同一种商品目录快照，现有列表、批量操作和刷新逻辑保持不变。

## 已验证事实

- Noon Catalog 登录页面使用内部接口 `POST /_vs/mp/mp-noon-catalog-api-rocket/offer/list/noon` 获取商品列表。
- 请求体包含 `page`、`per_page`、对象类型的 `filters`、`noon_store_code`、`sort` 和 `direction`。
- `per_page=100` 可返回 100 条；超过 100 会回退为 20 条。
- 真实店铺总数 109337 时，第 99 页仍返回 100 条，但第 100 页及之后固定返回 `total=0`、空列表；旧列表接口存在约 9900 条分页窗口，超过该范围必须失败并提示使用导出同步，不能写入残缺快照。
- 店铺 `PRJ517205` 的实测总数为 109337，单页请求约 0.7 至 0.8 秒。
- `noon_catalog_globalcatalogexport` 的 `live` 小分区已真实完成创建、轮询、下载和解析；`no_global_offer` 曾持续约 15 分钟处于 PENDING，三状态完整目录的真实端到端合并仍待 Noon 报表完成后验证。

## 用户界面

商品目录同步区域保留现有 FBN/FBP 与 Global 模式切换，并将单个“同步 SKU”按钮替换为两个并列按钮：

- `API 快速同步`：启动内部列表接口同步。
- `导出同步`：启动现有报表导出同步。

现阶段 Noon Global Catalog Export 仅支持 Global 模式。选择 FBN/FBP 模式时保留“导出同步”入口但禁用，并明确提示“导出同步仅支持 Global 模式”；服务端同时拒绝 `source=export` 与 `mode=fbn` 的组合，防止旧客户端或直接调用静默写入错误快照。后续若接入真实的 FBN/FBP 导出数据源，可在不改变现有 `source` 请求字段的前提下开放该组合。

同步期间两个按钮均禁用，避免同一店铺重复启动目录任务。“重新同步”入口不再使用不明确的默认来源，改为同样明确的两个操作或移除重复入口；优先复用顶部同步区域，避免增加第三套控制。

进度区域继续复用现有样式。API 方案显示已完成页数、总页数和已读取商品数；导出方案继续显示创建导出、等待生成、下载和解析阶段。

## 请求与任务契约

`POST /api/noon-catalog-sync-jobs` 请求体增加必填字段：

```json
{
  "storeId": "PRJ517205",
  "mode": "global",
  "source": "internal_api"
}
```

`source` 仅接受：

- `internal_api`
- `export`

缺失或无效值返回 400 和中文错误信息，不静默选择默认方案。这样旧调用点必须明确迁移，不会因为默认值改变而误用慢导出或内部接口。

任务记录继续使用现有 `syncNoonCatalog` 类型，通过日志和启动参数区分来源。任务响应和轮询接口不增加新的持久化结构。

## API 快速同步数据流

1. 任务使用所选店铺的 `.noon-profiles/<storeId>` 专用浏览器资料启动无头浏览器。
2. 打开该项目的 Catalog 页面，若跳转登录页则立即失败并提示先完成店铺登录。
3. 调用 `noon-store/list` 获取当前项目可用的 `noon_store_code`，不在代码中写死 `STR517205-NSA`。
4. 请求第一页 `offer/list/noon`，固定 `per_page=100`，读取 `data.total` 与 `data.hits`。
5. 根据总数计算页数，以最多 8 个并发请求读取剩余页面；任一页总数变化或触发第 100 页结果窗口时终止，不写入残缺快照。
6. 每完成一批页面输出结构化进度日志；单页失败即终止任务并保留明确错误，不自动切换来源。
7. 去重后把 `hits` 转换成现有快照 `rows` 结构，写入 `exports/noon-catalog-sync`。
8. 关闭浏览器资料，任务进入完成状态，页面沿用现有刷新流程读取新快照。

内部响应字段映射：

- 标题：优先从 `content` 中读取英文或可用标题。
- PSKU：`partner_sku`，为空时使用 `psku_code`。
- SKU：`zsku_child`，为空时使用 `catalog_sku`。
- 价格：`price` 与 `currency` 组合。
- 库存：组合 `fbn_stock`、`fbp_stock` 中可用数值。
- 问题与状态：使用 `offer_issues`、`live_status`、`seller_status` 的可读值。
- 图片：从 `content` 中可用的主图字段读取；没有则留空。

字段解析必须容忍字段缺失，但不得吞掉 HTTP 错误、登录失效或响应结构完全不符。

## 导出同步数据流

`scripts/sync-noon-catalog-api.js` 和 `scripts/lib/noon-catalog-api-sync.js` 负责导出：`noon_status=all` 在内部展开为 Noon 当前支持的 `live`、`not_live`、`no_global_offer` 三个互斥导出，依次创建、轮询、下载 CSV，以 PSKU/SKU 组合合并去重后写入相同快照目录。CSV 表头、列数、引号结构及稳定 SKU 标识不合法时必须失败，不得写成功快照。保留现有 `source=export` 与 `noonStatus=all` 请求结构兼容。

除接收明确的 `source=export` 任务路由外，不重构其认证、解析或文件格式。

## 并发与性能

- 单页固定 100 条。
- 并发上限固定为 8，不增加用户配置，避免投机性复杂度。
- 首次请求用于确认总数，后续页面按批次并发。
- 不无限重试；仅 HTTP 429 最多尝试 5 次并按 1、2、4、8 秒退避，其他网络或服务错误直接失败并提示用户可选择导出同步。
- 109337 条商品约需 1094 页。按实测请求耗时，并发 8 的网络读取阶段目标约 2 至 3 分钟，实际时间受 Noon 限流和浏览器环境影响。

## 错误处理与安全

- 本项目当前限定为本机单用户工具：Next 开发与生产启动只绑定 `127.0.0.1`；同步任务创建、任务读取和快照读取路由必须校验 loopback Host，写请求还必须校验同源 loopback Origin。不得直接改为局域网或公网监听；未来远程或多人使用时，需把共享本地访问守卫替换为真实身份认证和店铺级授权。
- 不输出 Cookie、Authorization、浏览器存储或 APIJWT 凭据。
- 店铺浏览器资料被占用时，沿用现有明确错误提示。
- 登录失效、店铺代码缺失、接口非 2xx、响应缺少 `data.total` 或 `data.hits` 均使任务失败。
- API 方案失败后不自动启动导出，避免用户误以为仍在快速同步。
- 两种方案只读取 Noon 商品数据并写本地快照，不执行 Noon 侧写操作。

## 文件边界

- 新增内部接口同步库和启动脚本，独立于现有导出实现。
- 修改 `src/lib/jobs.ts`，校验 `source` 并路由到对应脚本。
- 修改 `/noon-workbench` 客户端，提供两个明确按钮并传递 `source`。
- 更新现有测试或新增聚焦测试，不重构无关批量操作代码。

## 验收条件

- 页面可分别启动 API 快速同步和导出同步。
- 后端拒绝缺失或未知 `source`。
- API 同步动态取得 `noon_store_code`，以每页 100、最多 8 并发读取全部页面。
- API 响应能转换为现有快照格式，页面无需新的读取分支。
- API 进度可见，失败信息可见，且不会自动切换导出。
- 导出同步现有行为保持可用。
- 相关单元测试、类型检查和 Next.js 构建通过。
- 使用本地已登录测试店铺至少验证一次 API 快速同步的真实请求路径；若完整 10 万条验证耗时过长，至少验证第一页、总数、第二页和快照字段映射，并明确记录未执行的完整同步。

## 不做事项

- 不新增数据库或迁移。
- 不新增第三方依赖。
- 不开放并发数、页大小等高级配置。
- 不把内部接口认证信息复制到环境变量或前端。
- 不修改商品批量编辑、删除、上传或导出 Global 表功能。
- 不自动 commit、push、部署或操作生产数据库。
